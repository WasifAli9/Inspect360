Inspect360 AWS Migration Plan

This document describes the plan to migrate Inspect360 from a virtual Windows server to AWS with a minimal-cost setup: one EC2 server, one RDS database, and one S3 bucket for file storage (images, documents, and voice recordings). No load balancer in the initial phase; it can be added later when you need to scale.

Current State (Summary)

  • Backend: Node.js application running on one server.
  • Database: PostgreSQL.
  • File storage: All files (images, documents, voice) are stored on the server’s local disk.
  • Mobile app: Connects to the same backend using a configurable URL.

Target Architecture (Min-Cost Start)

  • Users and mobile app connect directly to a single EC2 instance (t3.micro).
  • EC2 runs the Inspect360 application and talks to RDS PostgreSQL (db.t3.micro) and one S3 bucket.
  • One S3 bucket is used for all file storage, with logical “folders” (prefixes) for images, documents, and voice. The application will be updated to store and retrieve files in these categories.
  • No Application Load Balancer initially; HTTPS is handled on the EC2 instance (e.g. with Let’s Encrypt). You can add a load balancer later for scaling and centralized HTTPS.

Phase 1: AWS Account and Prerequisites

  1. Create an AWS account at aws.amazon.com if you do not have one.
  2. Install the AWS CLI and configure it with your Access Key, Secret Key, and region (e.g. us-east-1).
  3. Choose one region (e.g. us-east-1) and use it for EC2, RDS, and S3.
  4. Domain: If you use a custom domain (e.g. portal.inspect360.ai), you can keep it where it is. If your domain is at GoDaddy: use GoDaddy’s DNS; you do not need Route 53. In GoDaddy, add an A record for your subdomain (e.g. portal) pointing to your EC2 Elastic IP. For HTTPS, use Let’s Encrypt on the EC2 instance (this works with GoDaddy-hosted domains as long as the A record points to that server).

Phase 2: Create One S3 Bucket

  1. In AWS Console → S3 → Create bucket, create one bucket with a globally unique name (e.g. inspect360-storage plus a suffix).
  2. Block all public access; files will be served through your application or signed URLs.
  3. Plan for up to 50 GB storage; S3 scales automatically. Budget about $1–2/month for 50 GB in us-east-1.
  4. Enable encryption (SSE-S3 is the default). Versioning is optional.
  5. Create an IAM user (or use an existing one) for the application with permission to get, put, delete objects and list the bucket. Save the Access Key and Secret securely; these will be used by the app to access S3.

Phase 3: RDS PostgreSQL (db.t3.micro)

  1. In RDS → Create database, choose PostgreSQL (match your current major version, e.g. 15).
  2. Use the Free tier or Dev/Test template. Instance: db.t3.micro. Storage: 20 GB (minimum). Single-AZ for lowest cost.
  3. Set a master username and password and store them securely. Do not enable public access; the database will be reached only from your EC2 instance in the same VPC.
  4. Create a security group that allows inbound traffic on port 5432 only from the EC2 instance’s security group.
  5. After creation, note the RDS endpoint hostname. Your application will use this in its database connection string (with the database name, username, password, and SSL).
  6. Create the database (e.g. inspect360) if not created by the wizard, then run your database migrations (from your machine or from EC2 once the app is deployed).

Phase 4: EC2 (t3.micro) for the Application

  1. Launch instance: Use Amazon Linux 2023 or Ubuntu 22.04. Instance type: t3.micro (free tier eligible). Create or select a key pair for SSH. Root volume: 8–20 GB.
  2. Security group: Allow inbound SSH (port 22) from your IP, and HTTP (80) and HTTPS (443) from the internet (0.0.0.0/0) so users can reach the app. Allow all outbound traffic (for RDS, S3, Stripe, Resend, OpenAI, etc.).
  3. Allocate an Elastic IP and attach it to the instance so the public IP does not change after reboot. You will point your domain (e.g. in GoDaddy) to this IP.
  4. Attach an IAM role to the EC2 instance with S3 permissions (or use IAM user keys in the application configuration).
  5. HTTPS and DNS: Configure HTTPS on the EC2 instance (e.g. nginx or Caddy as reverse proxy with Let’s Encrypt). If your domain is at GoDaddy, in GoDaddy DNS add an A record (e.g. portal or @) with the EC2 Elastic IP. No Route 53 needed. You can add an Application Load Balancer later if you want centralized TLS and scaling.

Phase 5: Application and Storage Updates (Summary)

The application will need to be updated so that:

  • It can use either local disk or S3 for file storage, controlled by configuration.
  • When using S3, it stores files in one bucket with separate “folders” (prefixes) for images, documents, and voice.
  • Upload and download flows use the same categories so that existing features (photos, documents, voice notes) continue to work, with files stored in S3 under the correct category.
  • Configuration (e.g. database URL, S3 bucket name, and app URL) is set via environment variables on the EC2 instance.
  • The mobile app is configured with the new public URL (e.g. https://portal.inspect360.ai) so it points to the EC2 server.

Detailed code and file changes are documented separately for developers.

Phase 6: Migration Steps (Order of Operations)

  1. Create the S3 bucket and IAM user/role (Phase 2).
  2. Create RDS and run database migrations. Do not point the live Windows app to it yet (Phase 3).
  3. Implement and test S3 storage and categories in the application (e.g. locally or in a test environment) so uploads and downloads work with the single bucket and image/document/voice prefixes.
  4. Export data from the current Windows server: Export the database to a backup file. Copy all stored files (images, documents, voice) from the server to a safe location (e.g. your laptop or a temporary S3 location).
  5. Import the database to RDS: Restore the backup into the RDS database. Fix any schema or extension issues if they appear.
  6. Migrate existing files to S3: Run a one-time process that uploads each file to the S3 bucket under the correct category (images, documents, or voice) based on file type. Optionally update database references to use the new path format.
  7. Deploy the application to EC2: Install Node.js, copy the application code, install dependencies, build, and configure environment variables (database URL, S3 settings, base URL, session secret, etc.). Start the application with a process manager (e.g. pm2 or systemd) so it keeps running and listens on all interfaces.
  8. Configure DNS: Point your domain to the EC2 instance (Elastic IP or public IP). If using GoDaddy, add an A record for your host (e.g. portal) with the EC2 Elastic IP. Verify that HTTPS works (reverse proxy and Let’s Encrypt) and that the app responds.
  9. Smoke test: Log in, upload an image, a document, and a voice recording, open an existing inspection with photos, and run a report to confirm everything works.
  10. Cutover: Update the mobile app configuration so it uses the new URL. Release an app update if needed. After a short period of parallel running, retire the Windows server.

Phase 7: Scaling Later (When Traffic Grows)

  • Backend: Add an Application Load Balancer, then put EC2 behind an Auto Scaling Group (multiple instances) using the same image and configuration.
  • Database: Resize RDS to a larger instance (e.g. db.t3.large) and optionally add read replicas for read-heavy workloads.
  • S3: The single bucket with prefixes can stay as is. You can optionally add CloudFront in front of S3 for faster delivery. If you later need different lifecycle or permissions per type, you can introduce additional buckets.

Security and Cost Reminders

  • Do not commit secrets (e.g. database URLs, API keys) to version control. Use IAM roles for EC2 where possible; rotate keys if they are ever exposed.
  • RDS: Keep public access off. Only the application (EC2) security group should be able to connect on port 5432.
  • S3: Block public access. Access to files should go through the application or time-limited signed URLs issued by the app.
  • Set a budget alert in AWS Billing so you are notified before costs exceed your expected range.
